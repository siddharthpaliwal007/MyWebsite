IAM — Identity and Access Management
Service #01 · Identity & Access Domain
6 criteria Critical ▼
Service Overview

AWS IAM is the foundational access control layer for every AWS resource. Misconfigurations here can lead to complete account compromise, privilege escalation, and lateral movement across the entire infrastructure. It is considered the highest-risk service in any AWS security audit.

Security Criteria Checked
  • Root account usage and active access keys on rootHIGH
  • MFA enforcement on all IAM users and root accountHIGH
  • Password policies — complexity, length, rotation, reuseMED
  • Excessive permissions / wildcard (*) actions in policiesHIGH
  • Unused IAM users and stale access keys (90+ days)MED
  • IAM role trust policies and cross-account access validationMED
Risk Assessment
Confidentiality ImpactCritical
Integrity ImpactCritical
Availability ImpactHigh
Full account takeover via root compromise
Data exfiltration through over-privileged roles
Lateral movement across AWS services
Remediation Steps
1
Delete or deactivate root account access keys immediately
2
Enable MFA on root and all privileged IAM users
3
Apply least privilege — replace * permissions with specific actions
4
Rotate or delete access keys older than 90 days
5
Enable IAM Access Analyzer and review trust policies

CloudTrail
Service #02 · Logging & Monitoring Domain
4 criteria Critical ▼
Service Overview

CloudTrail records every API call made across your AWS account. Without it, there is no forensic trail for incident response. Disabling or misconfiguring CloudTrail is a primary tactic used by attackers after account compromise to cover their tracks.

Security Criteria Checked
  • CloudTrail logging enabled in all AWS regionsHIGH
  • Multi-region trail configuration to prevent blind spotsMED
  • Log file validation enabled (tamper detection)HIGH
  • CloudWatch integration for real-time API activity monitoringMED
Risk Assessment
Audit & ForensicsCritical
Incident Response CapabilityCritical
No visibility into unauthorized API calls
Unable to detect privilege escalation post-breach
Compliance violations (PCI-DSS, SOC2, HIPAA)
Remediation Steps
1
Enable a multi-region CloudTrail trail covering all regions
2
Enable log file validation with SHA-256 digest files
3
Deliver logs to S3 with SSE-KMS encryption
4
Set up CloudWatch alarms for sensitive API calls (e.g., DeleteTrail)

Amazon S3 — Simple Storage Service
Service #03 · Data & Storage Domain
5 criteria Critical ▼
Service Overview

S3 is the most common source of cloud data breaches due to publicly exposed buckets. It stores everything from application data to backups and PII. A single misconfigured bucket can expose terabytes of sensitive information to the public internet without authentication.

Security Criteria Checked
  • Block Public Access settings — account and bucket levelHIGH
  • Default encryption (SSE-S3 or SSE-KMS) for all bucketsHIGH
  • Bucket policies — no overly permissive Principal: *MED
  • S3 server access logging enabled per bucketLOW
  • Versioning enabled for sensitive/critical bucketsLOW
Risk Assessment
Data Exposure RiskCritical
Mass PII/PCI data exposure via public bucket
Unencrypted data at rest violates compliance
Data deletion without versioning is unrecoverable
Remediation Steps
1
Enable S3 Block Public Access at the account level
2
Enable SSE-KMS default encryption on all buckets
3
Review and restrict bucket policies — deny Principal: *
4
Enable versioning and MFA delete on critical buckets

EC2 — Elastic Compute Cloud
Service #04 · Compute Domain
4 criteria High ▼
Service Overview

EC2 instances are the primary compute workhorses. Misconfigured instances with public IPs, over-privileged instance roles, or metadata service exploitation (SSRF attacks) can expose entire account credentials. IMDSv2 enforcement is one of the most critical EC2 security controls.

Security Criteria Checked
  • IMDSv2 enforced on all instances (prevent SSRF exploitation)HIGH
  • Public IP exposure on sensitive instancesMED
  • IAM instance profiles — least privilege role policiesHIGH
  • EBS volumes encrypted at launchMED
Risk Assessment
SSRF / Metadata ExploitCritical
IMDSv1 enables SSRF → credential theft via metadata endpoint
Over-privileged roles = account-wide blast radius
Unencrypted volumes expose data in snapshots
Remediation Steps
1
Enforce IMDSv2 on all instances via AWS Config rule or instance attribute
2
Remove public IPs from backend/database instances
3
Scope instance IAM roles to minimum required actions/resources
4
Enable EBS encryption by default in EC2 settings

Security Groups
Service #05 · Network & Edge Domain
2 criteria Critical ▼
Service Overview

Security Groups act as virtual firewalls for EC2 instances and other resources. Open ports to the internet (0.0.0.0/0) — especially SSH (22), RDP (3389), and database ports — are one of the most common attack surfaces in AWS and the leading cause of ransomware incidents.

Security Criteria Checked
  • SSH (22), RDP (3389) not open to 0.0.0.0/0 or ::/0CRITICAL
  • No overly permissive inbound rules allowing all trafficHIGH
Risk Assessment
Network ExposureCritical
Brute-force attack surface on SSH/RDP
Database ports (3306, 5432, 1433) exposed to internet
Ransomware entry vector via open RDP
Remediation Steps
1
Remove all 0.0.0.0/0 inbound rules on ports 22 and 3389
2
Use AWS Systems Manager Session Manager instead of SSH
3
Restrict all inbound traffic to known IP ranges or VPC CIDR blocks
4
Use AWS Config rule: restricted-ssh, restricted-common-ports

VPC — Virtual Private Cloud
Service #06 · Network & Edge Domain
4 criteria High ▼
Service Overview

VPCs define the network boundary for your AWS resources. Misconfigured VPCs — with missing flow logs, improper subnet design, or insecure routing — can allow attackers to move laterally within your infrastructure without detection.

Security Criteria Checked
  • VPC Flow Logs enabled for all VPCs and subnetsHIGH
  • No use of default VPC for production resourcesHIGH
  • Private vs public subnet segmentation for sensitive resourcesMED
  • Route table review — no unintended internet gateway routesMED
Risk Assessment
Network VisibilityHigh
No flow logs = blind to lateral movement and exfiltration
Default VPC has permissive defaults enabling exposure
Flat network allows east-west attacker movement
Remediation Steps
1
Enable VPC Flow Logs for all VPCs, deliver to S3 or CloudWatch
2
Migrate workloads out of the default VPC
3
Segment workloads into public/private/isolated subnets
4
Audit route tables for unexpected 0.0.0.0/0 routes

RDS — Relational Database Service
Service #07 · Data & Storage Domain
4 criteria Critical ▼
Service Overview

RDS instances often contain the most sensitive data in an organization — customer records, financial data, authentication credentials. Public accessibility combined with weak credentials or unencrypted storage is among the most severe misconfiguration patterns.

Security Criteria Checked
  • Storage encryption enabled using KMS CMKsHIGH
  • Public accessibility disabled (no publicly accessible endpoint)CRITICAL
  • Automated backups enabled with appropriate retention periodMED
  • Snapshot encryption and cross-region copy settingsMED
Risk Assessment
Data Breach RiskCritical
Publicly accessible RDS = direct attack surface for DB exploits
Unencrypted snapshots shared accidentally expose full DB dump
No backups = ransom-viable target with no recovery
Remediation Steps
1
Disable public accessibility on all RDS instances
2
Enable storage encryption (requires snapshot restore to new encrypted instance)
3
Set automated backup retention to at least 7 days
4
Encrypt all manual snapshots before sharing cross-account

AWS Lambda
Service #08 · Compute Domain
3 criteria Medium ▼
Service Overview

Lambda functions execute code in response to events and often have IAM roles that interact with many AWS services. Over-privileged execution roles, missing logging, and sensitive data in environment variables are common security gaps in serverless architectures.

Security Criteria Checked
  • CloudWatch Logs / X-Ray tracing enabled for observabilityMED
  • Execution role permissions scoped to least privilegeHIGH
  • CloudTrail coverage for Lambda invocations and data eventsLOW
Remediation Steps
1
Enable CloudWatch Logs for all Lambda functions
2
Apply unique per-function IAM roles with minimum permissions
3
Store secrets in Secrets Manager, not environment variables
Risk Assessment
Over-privileged role = entire account accessible from Lambda exploit
No logs = zero forensic visibility for serverless attacks
Hardcoded secrets in env vars = credential theft via code access

Audit all EBS snapshots for public sharing via AWS CLI or Config

Enable EBS default encryption in all regions
3
Use AWS Config rule: ebs-snapshot-public-restorable-check

Public snapshot = complete disk image accessible by anyone
Credentials, private keys, application data all exposed
Unencrypted volumes violate data-at-rest compliance requirements

CloudWatch is the central observability platform. Without proper metric filters and alarms on security-critical events (root login, unauthorized API calls, security group changes), threats can persist undetected for days or weeks.

Security Criteria Checked

  • Unauthorized API calls
  • Security group changes
  • IAM policy changes

Remediation Steps
1
Create CloudWatch metric filters on CloudTrail log group for all CIS alarms
2
Connect alarms to SNS topics for real-time notification
3
Set log group retention to 90 days minimum for compliance

2 criteria High ▼
Service Overview

KMS manages encryption keys used across S3, EBS, RDS, and other AWS services. Without key rotation, compromised keys remain valid indefinitely. Poor key policies can allow unauthorized decryption of sensitive data even after access is revoked at the IAM level.

Security Criteria Checked
  • Automatic key rotation enabled for all customer-managed keys (CMKs)HIGH
  • Key policies reviewed — no overly permissive kms:* grantsHIGH
Remediation Steps
1
Enable automatic annual key rotation on all CMKs
2
Audit key policies using IAM Access Analyzer for KMS
3
Enable CloudTrail logging for all KMS API calls
Risk Assessment
No rotation = long-lived keys increase compromise window
Overly permissive key policy bypasses IAM access controls

AWS Secrets Manager
Service #16 · Encryption & Secrets Domain
2 criteria High ▼
Service Overview

Secrets Manager centralizes storage of database credentials, API keys, and other sensitive values. The primary risk is secrets that are never rotated, meaning a stolen secret remains valid indefinitely. Secrets stored outside Secrets Manager (in code, env vars, or S3) are also flagged.

Security Criteria Checked
  • Automatic rotation enabled on all secretsHIGH
  • Secrets not stored in plaintext (env vars, source code, S3)HIGH
Remediation Steps
1
Enable automatic rotation for all RDS, Redshift, and API secrets
2
Migrate any plaintext secrets from env vars or config files
3
Use Macie to scan S3 for accidentally stored credentials
Risk Assessment
Unrotated secrets remain exploitable indefinitely after breach
Hardcoded secrets in code repos are frequently leaked on GitHub

Service Overview

ECS and EKS orchestrate container workloads. Security posture gaps include running containers in privileged mode, using host network namespaces, assigning over-privileged task roles, and failing to enforce Pod Security Standards or OPA policies for Kubernetes.

Security Criteria Checked
  • No containers running in privileged modeHIGH
  • Task/pod IAM roles scoped to least privilegeHIGH
Remediation Steps
1
Disable privileged mode in all task definitions and pod specs
2
Use IRSA (IAM Roles for Service Accounts) in EKS with minimal scope
3
Enable EKS control plane logging (API, audit, authenticator)

ELB / ALB / NLB — Load Balancers
Service #23 · Network & Edge Domain
2 criteria High ▼
Service Overview

Load balancers are the primary entry point for web traffic. Weak TLS policies (allowing TLS 1.0/1.1, weak cipher suites) expose traffic to downgrade attacks. HTTP listeners without HTTPS redirect expose credentials and session tokens transmitted in plaintext.

Security Criteria Checked
  • TLS 1.2+ enforced — TLS 1.0/1.1 disabled on all listenersHIGH
  • HTTP listeners redirect to HTTPS (port 443)MED
Remediation Steps
1
Update security policy to ELBSecurityPolicy-TLS13-1-2-2021-06 or newer
2
Add HTTP → HTTPS redirect rules on port 80 listeners
3
Enable ALB access logs to S3 for traffic visibility
Risk Assessment
Weak TLS allows POODLE/BEAST downgrade attacks
HTTP traffic = plaintext credentials and session tokens

ACM manages SSL/TLS certificates for AWS services. Expired certificates cause browser security warnings, break HTTPS connections, and can signal certificate mismanagement. Unused certificates represent attack surface and should be removed.

Security Criteria Checked
  • No expired or soon-to-expire certificates in useHIGH
  • All ACM certificates in active use — unused certs cleaned upLOW
Remediation Steps
1
Use ACM-managed certs (auto-renew) instead of imported certificates
2
Set up CloudWatch alarm for certificate expiry within 30 days
3
Delete unused certificates to reduce attack surface
Risk Assessment
Expired certs break HTTPS, enabling MitM attacks
Imported certs don't auto-renew → production outage risk

2 criteria High ▼
Service Overview

AWS Backup provides centralized backup management across EBS, RDS, DynamoDB, EFS, FSx, Aurora, and Storage Gateway. Without structured backup plans and tested recovery procedures, ransomware or accidental deletion incidents become unrecoverable disasters.

Security Criteria Checked
  • Backup plans configured for all critical resources with RPO/RTO targetsHIGH
  • Cross-region or cross-account backup copies for disaster recoveryHIGH
Remediation Steps
1
Create backup plans covering RDS, EBS, DynamoDB, and EFS resources
2
Configure cross-region copy rules for critical backups
3
Enable Backup Vault Lock to prevent backup deletion by ransomware
4
