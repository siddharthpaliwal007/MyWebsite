Identity & Access Management

AWS IAM
Identity and Access Management
aws.amazon.com/iam
▾
Overview

AWS Identity and Access Management (IAM) is the foundational security service in AWS that enables you to securely manage access to AWS services and resources. IAM lets you control who is authenticated (signed in) and what they are authorized to do. At its core, IAM operates on the principle of least privilege — granting only the permissions needed to perform a task and nothing more. IAM is a global service, meaning its configurations apply across all AWS regions, making it the single control plane for all access decisions in your AWS environment.

Key Features
👤 Users & Groups
📋 IAM Policies (JSON)
🎭 IAM Roles
🔑 MFA Support
🔗 Federation (SSO/SAML)
🔒 Permission Boundaries
⚙️ Service Control Policies
📊 Access Analyzer
How It Works
1
A Principal (user, role, or service) makes a request to access an AWS resource.
2
AWS authenticates the principal using their credentials (password, access keys, or temporary tokens).
3
IAM evaluates all applicable policies — identity-based, resource-based, permission boundaries, and SCPs.
4
Access is allowed or denied based on the evaluation result. Explicit deny always wins.
💡 Best Practice: Avoid using the root account for daily tasks. Create individual IAM users, attach policies, and enable MFA on all privileged accounts.
🌍 Real-World Scenario
Netflix — Preventing Credential Misuse & Privilege Escalation
⚔ Attack: Privilege Escalation

Netflix enforces strict IAM role-based access across hundreds of microservices running on AWS. By applying fine-grained IAM policies per service and using IAM roles with short-lived credentials (via STS AssumeRole), Netflix prevents lateral movement and privilege escalation attacks. If a container is compromised, its IAM role only has access to its specific S3 bucket or DynamoDB table — not the entire account — drastically limiting blast radius.

IAM Identity Center
formerly AWS Single Sign-On (SSO)
aws.amazon.com/iam/identity-center
▾
Overview

IAM Identity Center is AWS's centralized identity management service that enables single sign-on (SSO) access across all your AWS accounts and cloud applications from one place. It integrates with external identity providers (IdPs) such as Microsoft Active Directory, Azure AD, Okta, and Google Workspace using SAML 2.0, allowing employees to use their existing corporate credentials to access AWS resources. Rather than managing separate IAM users in each AWS account, Identity Center provides a single pane of glass for access management across multi-account AWS Organizations environments.

Key Features
🚪 Single Sign-On
📂 Multi-Account Access
🔌 SAML 2.0 / OIDC
🗂 Permission Sets
👥 AD / Azure AD Integration
📱 MFA Enforcement
📊 Access Logging
🌐 App Assignments
How It Works
1
Connect your existing corporate identity provider (Okta, Azure AD, etc.) to IAM Identity Center.
2
Define Permission Sets that map to IAM roles (e.g., "Developer", "ReadOnly", "Admin").
3
Assign groups/users to specific AWS accounts with the appropriate permission set.
4
Users log in once via the Identity Center portal and get temporary credentials for any assigned account.
🌍Real-World Scenario
Airbnb — Securing Access Across 100+ AWS Accounts
⚔ Attack: Account Takeover via Compromised Corporate Credentials

Airbnb manages dozens of AWS accounts across dev, staging, and production environments. By federating through IAM Identity Center with Okta as the IdP, any compromised employee credential triggers automatic session revocation from a single control point. MFA is enforced at the IdP layer, so even phished passwords alone cannot grant AWS access, preventing the account takeover attacks that have hit many cloud-native companies.

Amazon Cognito
Customer Identity & Access Management (CIAM)
aws.amazon.com/pm/cognito
▾
Overview

Amazon Cognito is a fully managed customer identity platform designed for web and mobile applications. Unlike IAM (which manages internal AWS users), Cognito manages your end customers — enabling sign-up, sign-in, and access control for millions of users. It provides two main components: User Pools (a user directory for authentication) and Identity Pools (for granting temporary AWS credentials to authenticated users). Cognito supports social sign-in (Google, Facebook, Apple), enterprise SAML/OIDC federation, and built-in security features like adaptive authentication, compromised credential detection, and advanced security mode.

Key Features
📧 User Pools (Auth)
🔑 Identity Pools (AuthZ)
🌐 Social Login (Google/FB)
📱 MFA & OTP
🤖 Bot Detection
🔐 JWT Tokens
🔄 SAML / OIDC
📊 Adaptive Auth
How It Works
1
User signs up/logs in through your app, which calls the Cognito User Pool API.
2
Cognito verifies credentials, performs MFA, and issues JWT tokens (ID, Access, Refresh).
3
The app exchanges tokens with the Identity Pool to get temporary AWS credentials (via STS).
4
User accesses AWS resources (e.g., S3, API Gateway) with scoped, temporary credentials.
🌍Real-World Scenario
Samsung SmartThings — Blocking Credential Stuffing at Scale
⚔ Attack: Credential Stuffing / Account Takeover

SmartThings, Samsung's IoT platform built on AWS, uses Amazon Cognito with Advanced Security Mode enabled to protect millions of smart home user accounts. Cognito's built-in compromised credential checks and adaptive authentication automatically challenge suspicious login attempts — such as logins from unusual geolocations or via known breached password lists — blocking large-scale credential stuffing campaigns that attempted to hijack user accounts and control connected home devices.

AWS Secrets Manager
Secure credential & secret lifecycle management
aws.amazon.com/secrets-manager
▾
Overview

AWS Secrets Manager is a fully managed service that enables you to securely store, manage, and automatically rotate sensitive information such as database credentials, API keys, OAuth tokens, and third-party service passwords. It eliminates the dangerous practice of hardcoding secrets in code, configuration files, or environment variables. Secrets are encrypted at rest using AWS KMS and access is controlled via IAM policies. The service natively integrates with Amazon RDS, Redshift, and DocumentDB to rotate credentials automatically without application downtime, implementing zero-trust principles for secret management.

Key Features
🔄 Automatic Rotation
🔐 KMS Encryption
🗃 Versioning
📋 IAM-based Access Control
🔗 RDS / Aurora Integration
📡 Cross-Account Access
📊 Audit via CloudTrail
📦 Resource Tagging
How It Works
1
Store a secret (e.g., DB password) in Secrets Manager. It's encrypted with KMS immediately.
2
Application calls GetSecretValue API at runtime — no hardcoded credentials in code.
3
A Lambda rotation function fires on schedule, generates a new credential, and updates both the DB and the secret store.
4
All access attempts are logged to CloudTrail for audit purposes.
🌍Real-World Scenario
Capital One — Preventing Secrets Exposure from Source Code Leaks
⚔ Attack: Secrets Exfiltration via Git Commit / SSRF

Following the 2019 Capital One breach (where hardcoded credentials contributed to massive data exposure), the financial institution adopted AWS Secrets Manager to ensure zero credentials ever appear in source code or environment variables. Secrets rotate automatically every 30 days, and any access outside approved IAM roles triggers an immediate alert. This approach directly addresses the server-side request forgery (SSRF) attack vector that was exploited to steal credentials from EC2 instance metadata.

Amazon GuardDuty
ML-powered continuous threat detection
aws.amazon.com/guardduty

Overview

Amazon GuardDuty is an intelligent threat detection service that continuously monitors your AWS accounts, workloads, and data for malicious activity and unauthorized behavior. It analyzes billions of events across multiple AWS data sources — including AWS CloudTrail logs, VPC Flow Logs, and DNS logs — using machine learning, anomaly detection, and integrated threat intelligence (from AWS, CrowdStrike, and Proofpoint). GuardDuty identifies threats such as cryptocurrency mining, credential theft, data exfiltration, reconnaissance attacks, and compromised EC2 instances — all without requiring you to deploy or manage any security software or infrastructure.

3
Findings are generated with severity levels (Low/Medium/High) and actionable context.

Lyft — Detecting Cryptomining after EC2 Compromise
⚔ Attack: EC2 Cryptomining via Compromised Credentials

Lyft uses Amazon GuardDuty across its entire AWS organization to detect unauthorized resource usage. When attackers gained access to misconfigured IAM credentials and launched GPU-based EC2 instances for cryptocurrency mining, GuardDuty flagged the unusual API calls and abnormal network communication within minutes. An automated EventBridge → Lambda pipeline immediately quarantined the compromised instances and revoked the exposed credentials before significant damage occurred.

AWS CloudTrail
API activity logging & governance audit trail
aws.amazon.com/cloudtrail
▾
Overview

AWS CloudTrail is the AWS governance, compliance, and operational auditing service that records every API call and activity made in your AWS account. Every action — whether from the AWS Console, CLI, SDKs, or other services — generates an event log stored in S3 and optionally in CloudWatch Logs. CloudTrail answers the critical security questions: Who did what, when, and from where? It provides an immutable, tamper-evident log of all management events (control plane) and data events (data plane operations on S3, Lambda, DynamoDB). CloudTrail Insights uses ML to automatically detect unusual API activity patterns indicative of an attack.

AWS Config
Resource configuration history, change tracking & compliance
aws.amazon.com/config
▾
Overview

AWS Config is a service that enables you to assess, audit, and evaluate the configurations of your AWS resources over time. It continuously records the configuration state of every AWS resource in your account and tracks how configurations change. Config allows you to define Config Rules — desired configuration policies — and automatically evaluates resources against them. When a resource deviates from the desired state (e.g., an S3 bucket becomes public, a security group opens port 22 to the world), Config flags it as non-compliant and can trigger automated remediation via SSM Automation. Config is indispensable for cloud governance, change management, and regulatory compliance (SOC 2, PCI DSS, HIPAA).

AWS Key Management Service (KMS)
Managed encryption key creation & control
aws.amazon.com/kms
▾
Overview

AWS Key Management Service (KMS) is a fully managed encryption service that makes it easy to create, control, and manage cryptographic keys used to encrypt your data. KMS integrates seamlessly with nearly all AWS services (S3, EBS, RDS, DynamoDB, Lambda, Secrets Manager, etc.) to provide envelope encryption, where a Customer Master Key (CMK) encrypts data keys that in turn encrypt the actual data. Keys are protected by FIPS 140-2 validated Hardware Security Modules (HSMs). KMS provides a detailed audit log via CloudTrail of every key usage, ensuring you always know who used which key, when, and for what purpose.

Key Features
🔐 AES-256 / RSA / ECC
🏭 FIPS 140-2 HSMs
🔄 Automatic Key Rotation
📋 Key Policies
🌐 Cross-Region Replication
🔗 Multi-Region Keys
📊 CloudTrail Audit
🏢 Custom Key Stores (HSM)
How It Works — Envelope Encryption
1
Your app requests a data encryption key (DEK) from KMS. KMS generates a plaintext + encrypted copy of the DEK.
2
Your app uses the plaintext DEK to encrypt your data locally, then discards the plaintext DEK.
3
Only the encrypted DEK is stored alongside the ciphertext — never the plaintext key.
4
To decrypt, your app sends the encrypted DEK to KMS. KMS decrypts it using the CMK (after validating IAM + key policy), and returns the plaintext DEK.
🌍Real-World Scenario
Johnson & Johnson — Protecting Clinical Trial Data from Ransomware
⚔ Attack: Ransomware / Data Theft

Johnson & Johnson's pharmaceutical division stores sensitive FDA-regulated clinical trial data on AWS. By enforcing KMS encryption on all S3 buckets and RDS databases with customer-managed CMKs, even if ransomware actors exfiltrate raw data, it remains completely unreadable without the CMK — which is controlled by strict IAM key policies accessible only to a small set of audited principals. Annual key rotation and CloudTrail key usage logging ensure full traceability of every data access event.

AWS WAF
Web Application Firewall — layer 7 traffic filtering
aws.amazon.com/waf
▾
Overview

AWS WAF (Web Application Firewall) is a managed web application firewall that protects your web applications and APIs from common web exploits and bots that may affect availability, compromise security, or consume excessive resources. WAF operates at Layer 7 (Application Layer) and can be deployed in front of Amazon CloudFront, Application Load Balancers (ALB), Amazon API Gateway, and AWS AppSync. You define Web ACLs containing rules that inspect HTTP/HTTPS request components — headers, URI, query strings, cookies, and body — and either allow, block, or count requests that match. AWS Managed Rule Groups provide pre-built protections against OWASP Top 10, known bot signatures, and IP reputation lists.

2
Rules are evaluated in priority order — each rule inspects request components and takes an action (Allow/Block/Count/CAPTCHA).
3
AWS Managed Rule Groups (e.g., AWSManagedRulesCommonRuleSet) provide turnkey protection with no rule writing needed.
4
WAF logs are streamed to Kinesis Firehose → S3 → Athena for detailed traffic analysis and tuning.

By deploying AWS WAF with the Bot Control managed rule group on their CloudFront distribution, Ticketmaster can distinguish between legitimate browsers and automated bots using device fingerprinting and behavioral analysis, blocking bot traffic while allowing real users through — dramatically reducing fraudulent ticket purchases and protecting customer accounts.

Security Groups
Stateful virtual firewall for EC2 instances & resources
docs.aws.amazon.com/vpc/…/security-groups
▾
Overview

Security Groups are the foundational network security control in AWS VPC — they act as a stateful virtual firewall at the instance/resource level (Layer 3/4), controlling inbound and outbound traffic. Every EC2 instance, RDS database, Lambda function in a VPC, EKS node, and load balancer is associated with one or more security groups. Security groups are allow-only (no explicit deny rules) — any traffic not explicitly permitted is automatically denied. Being stateful means return traffic for allowed connections is automatically permitted without needing a matching outbound rule. Security groups work alongside NACLs (Network Access Control Lists), which provide stateless, subnet-level filtering as a defense-in-depth layer.

How It Works
1
Create a Security Group within a VPC, defining inbound rules (what traffic is allowed in) by protocol, port, and source (CIDR or another SG).
2
Define outbound rules controlling what traffic leaves. By default, all outbound traffic is allowed.
3
Associate the SG with your EC2 instance, RDS, or ALB. Rules take effect immediately and are evaluated at the hypervisor level.
4
Use SG-to-SG references for micro-segmentation — e.g., only allow your web tier SG to talk to your database tier SG on port 5432.
💡 Defense-in-Depth: Combine Security Groups (instance-level, stateful) with Network ACLs (subnet-level, stateless) for layered network security. Also use AWS Network Firewall for advanced L7 VPC inspection.
🌍Real-World Scenario
Stripe — Micro-segmentation Preventing Lateral Movement
⚔ Attack: Lateral Movement / East-West Network Attack

Stripe's PCI DSS-compliant payment processing infrastructure on AWS uses granular Security Group rules to enforce strict micro-segmentation between service tiers. Each microservice has its own dedicated Security Group that only permits traffic from the specific upstream services that need to communicate with it — no broad "all internal traffic" rules. When a pen test revealed a compromised Lambda function, the Security Group rules prevented any lateral movement to adjacent payment processing services, completely containing the blast radius to a single isolated microservice.
